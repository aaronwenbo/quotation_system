# 产品报价统一核心架构说明

## 问题背景

之前 Web 应用和 CLI 技能有各自独立的匹配逻辑实现，导致：
1. **结果不一致**：相同文件处理得到不同的匹配数（Web: 97 vs 技能: 108）
2. **重复维护**：匹配逻辑变更需要同时修改两处代码
3. **容易出错**：不同实现的细节差异难以追踪（如 Web 缺少规则四、规则五）

## 解决方案

创建**单一的核心库** (`product_quoting_core/`)，所有上层应用共享相同的匹配逻辑。

### 架构图

```
┌───────────────────────────────────────────────────────────┐
│                     product_quoting_core                   │
│  ┌───────────┐  ┌───────────┐  ┌───────────────────────┐  │
│  │ library   │  │ matcher   │  │ service               │  │
│  │ (标准库)  │  │ (5条规则) │  │ (统一报价服务)        │  │
│  └───────────┘  └───────────┘  └───────────────────────┘  │
└─────────────────────────────┬─────────────────────────────┘
                              │
           ┌──────────────────┴──────────────────┐
           │                                     │
┌──────────▼──────────┐              ┌──────────▼──────────┐
│   product_quoting_web  │          │ product-quoting (技能) │
│   (Flask Web应用)     │          │  (CLI/飞书机器人)      │
│   quoting_wrapper.py  │          │  core_compat.py        │
│   (薄包装层)          │          │  (兼容层)              │
└───────────────────────┘          └───────────────────────┘
```

### 目录结构

```
product_data_clean/
├── product_quoting_core/          # 统一核心库（唯一真实来源）
│   ├── __init__.py                 # 导出接口
│   ├── library.py                  # 标准产品库 + 规则配置
│   ├── matcher.py                  # 完整的5条匹配规则
│   └── service.py                  # 统一报价/更新服务
│
├── product_quoting_web/           # Web应用
│   ├── app.py                      # Flask路由
│   └── quoting_wrapper.py          # ← 仅导入，无业务逻辑
│
├── .claude/skills/product-quoting/ # 技能
│   ├── scripts/
│   │   ├── main.py                 # CLI入口
│   │   └── core_compat.py          # ← 兼容层，仅导入核心
│   └── data/                       # 技能独立的数据目录
│
└── docs/superpowers/统一核心架构说明.md  # 本文档
```

## 核心优势

✅ **结果一致性**：Web应用和技能使用完全相同的匹配逻辑，处理结果100%一致

✅ **单点维护**：匹配逻辑只需在 `product_quoting_core/` 中修改一次，所有上层应用自动生效

✅ **规则完整性**：核心库实现了完整的5条匹配规则：
   - 规则一：第五位1↔2双向互换
   - 规则二：末尾T移除（预处理变体）
   - 规则三：去*及后续内容（预处理变体）
   - 规则四：O→0字符容错
   - 规则五：前五位编码映射表

✅ **可扩展**：新增匹配规则或功能只需在核心库中实现

## 验证结果

使用 `接头询价(4.xls` 测试文件：

| 入口方式 | 直接匹配 | 规则一 | 规则二 | 规则四 | 规则五 | 总匹配 | 未匹配 |
|---------|---------|-------|-------|-------|-------|-------|-------|
| 核心库直接调用 | 93 | 1 | 3 | 3 | 8 | 108 | 187 |
| Web应用包装层 | 93 | 1 | 3 | 3 | 8 | 108 | 187 |
| 技能兼容层 | 93 | 1 | 3 | 3 | 8 | 108 | 187 |

**三者结果完全一致 ✓**

## 工作流程

### 修改匹配逻辑时

1. **仅修改** `product_quoting_core/matcher.py`
2. 测试验证
3. 提交代码
4. Web应用和技能自动获得更新，无需修改任何代码

### 更新标准库数据时

- Web应用：使用 `product_quoting_web/data/` 目录下的文件
- 技能：使用 `.claude/skills/product-quoting/data/` 目录下的文件
- 注意：两者数据独立，如需同步需手动复制

## 注意事项

1. **不要在包装层写业务逻辑**：`quoting_wrapper.py` 和 `core_compat.py` 只负责导入，不包含任何匹配逻辑
2. **数据文件独立性**：Web应用和技能可以使用各自的数据目录，也可以共享
3. **版本兼容**：核心库接口保持稳定，新增功能使用可选参数
