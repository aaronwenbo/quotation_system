# 产品编码自动报价系统 — Agent 技能编写指南

## 系统概述

一个产品编码智能匹配与自动报价系统。上传订单表格，系统通过5条匹配规则自动识别产品编码并填充价格，生成带颜色标注的报价结果。形成"报价→人工补价→更新标准库"的闭环流程。

- **标准库规模**：当前 ~2045 个产品,慢慢在增加
- **Web 应用地址**：`http://http://111.229.112.24:5002/`
- **认证方式**：HTTP Basic Auth（默认 `admin` / `mushroom123`）
- **支持格式**：`.xlsx`、`.xls`、`.csv`

---

## API 接口（Agent 需要调用的 HTTP 请求）

所有接口都需要 HTTP Basic Auth 认证头。

### 1. 上传订单文件 → 获取预览

```
POST /upload
Content-Type: multipart/form-data

参数:
  file: <订单文件>

返回: HTML 页面，包含:
  - job_id: 本次任务唯一ID
  - filename: 服务器保存的文件名
  - 前10行数据预览（表格形式）
  - 列号选择器（0-based: A=0, B=1, C=2...）
```

Agent 需要从返回的 HTML 中提取：
- `job_id` 和 `filename`（隐藏表单字段）
- 列预览数据，推断产品编码列和数量列的位置

### 2. 执行报价处理

```
POST /process
Content-Type: application/x-www-form-urlencoded

参数:
  job_id:       <从 /upload 获取>
  filename:     <从 /upload 获取>
  code_col:     <产品编码列号，0-based，如 0=A, 1=B>
  qty_col:      <数量列号，0-based>
  use_markup:   "on" 或 不传（是否启用价格上浮）
  markup_percent: <上浮百分比，如 10 表示上浮10%>

返回: HTML 页面，包含:
  - stats: 匹配统计（见下方结构）
  - no_match_codes: 未匹配编码列表
  - qty_warnings: 数量列解析警告（如有）
  - result_filename: 结果文件下载名
```

### 3. 下载报价结果

```
GET /download/<result_filename>

返回: Excel 文件（带颜色样式）
  - 绿色背景 → 直接匹配
  - 黄色背景 → 规则匹配
  - 红色背景 → 无匹配（需人工填写价格）
```

### 4. 上传反馈文件 → 获取预览

```
POST /upload-update
Content-Type: multipart/form-data

参数:
  file: <人工补价后的报价反馈文件>

返回: HTML 页面，包含预览和列选择器
```

### 5. 执行标准库更新

```
POST /process-update
Content-Type: application/x-www-form-urlencoded

参数:
  job_id:       <从 /upload-update 获取>
  filename:     <从 /upload-update 获取>
  code_col:     <产品编码列号>
  price_col:    <价格列号>
  label_col:    <匹配标注列号（筛选"无匹配"项）>
```

处理逻辑：自动备份当前标准库 → 筛选标注为"无匹配"且有价格的行 → 清洗编码后加入标准库 → 写入更新日志。最多保留最近5个备份。

### 6. 查看标准库状态

```
GET /library

返回: HTML 页面，包含:
  - 标准库产品总数
  - 备份文件列表（时间、大小、下载链接）
  - 最近50条更新日志（加入时间、标准编码、原始编码、价格）
```

### 7. 手动备份标准库

```
POST /backup-library

返回: 重定向到 /library，显示成功/失败消息
```

## 匹配规则详解（5条规则，优先级从高到低）

处理流程：首先生成预处理变体（原始编码、去T、去\*、去T+去\*、去\*+去T），再对每个变体依次尝试以下规则：

| 优先级 | 规则 | 说明 | 示例 |
|--------|------|------|------|
| 最高 | **直接匹配** | 编码与标准库完全一致 | `20411-16-05` → 命中 |
| ↓ | **规则一** | 去掉`-`后，第5位字符 `1`↔`2` 双向互换 | `20411-16-05` ↔ `20412-16-05` |
| ↓ | **规则五** | 前5位编码按映射表匹配（可配置） | `30411-xx-xx` → `20411-xx-xx` |
| ↓ | **规则四** | 字母 `O`/`o` 替换为数字 `0` | `2041O-16-05` → `20410-16-05` |
| — | **规则二** | 移除编码末尾的 `T`（预处理变体） | `20411-16-05T` → `20411-16-05` |
| — | **规则三** | 去掉 `*` 及其后面直到下一个 `-` 的内容（预处理变体） | `20411-16-05*XX-06` → `20411-16-05-06` |

一旦命中立即停止，返回匹配的变体 + 规则组合标签。

## 报价处理返回的 stats 结构

```python
{
    "total": 125,            # 处理的总行数（不含表头）
    "direct_match": 94,      # 直接匹配
    "rule1_match": 1,        # 规则一（1↔2互换）
    "rule2_match": 3,        # 规则二（去T）
    "rule3_match": 0,        # 规则三（去*）
    "rule4_match": 3,        # 规则四（O→0）
    "rule5_match": 8,        # 规则五（前缀映射）
    "rule_combination": 0,   # 规则组合（去T+去*等）
    "no_match": 16,          # 无匹配
    "qty_warnings": []       # 数量列警告列表
}
```

匹配率 = (total - no_match) / total × 100%

## Agent 技能建议工作流

### 场景一：用户上传订单要求报价

```
1. Agent 收到订单文件
2. POST /upload 上传文件
3. 从返回的 HTML 中解析：
   - 前10行预览数据 → 推断哪个列是产品编码、哪个列是数量
   - 提取 job_id 和 filename
4. 向用户确认：产品编码列和数量列是否正确
5. 询问是否需要价格上浮（可选）
6. POST /process 执行报价
7. 解析返回的统计信息，用简洁的格式告知用户：
   - 共处理 X 行，匹配 Y 行，匹配率 Z%
   - 各规则命中数量
   - 未匹配编码列表（如果 ≤20 个）
8. GET /download/<result_filename> 下载结果文件提供给用户
9. 提醒用户：对于"无匹配"的产品，在结果文件中填写价格后可以更新标准库
```

### 场景二：用户要求更新标准库

```
1. Agent 收到反馈文件（人工补价后的报价结果）
2. POST /upload-update 上传文件
3. 从预览中确认列号（产品编码列、价格列、标注列）
4. POST /process-update 执行更新
5. 告知用户：新增 X 个产品，跳过 Y 个（已存在）
```

### 场景三：用户查询标准库状态

```
1. GET /library 获取状态页
2. 解析并告知用户：
   - 当前标准库产品总数
   - 最近更新记录
   - 可用备份列表
```

## 列号推断提示

Agent 在解析文件预览时，可以通过以下特征推断列号：

- **产品编码列**：包含类似 `20411-16-05`、`30491-18-06` 格式的值（数字-数字-数字）
- **数量列**：纯数字，通常为整数，值范围在 1~10000
- **价格列**：小数，如 `28.305`、`35.955`
- **材料/规格列**：中文或英文文本，如 `SS316`
- **标注列**（更新时）：值为"直接匹配"、"规则一(1↔2)"、"规则二(去T)"、"无匹配"等

## 规则五扩展方式

如需新增前缀映射规则，编辑服务器上的 `product_quoting_web/data/matching_rules_config.json`：

```json
{
  "rule5_prefix_mapping": {
    "mappings": {
      "30411": ["20411", "30411→20411"],
      "新前五位": ["目标前五位", "说明"]
    }
  }
}
```

无需重启服务即可生效。

## 注意事项

1. 上传文件时务必确认列号（0-based：A=0, B=1, C=2...）
2. 如果文件第一行是表头，系统会自动跳过（`skip_rows=1`）
3. 标准库更新前会自动备份（最多保留5个备份）
4. 未匹配的产品需要人工判断：是新产品需要入库，还是编码格式异常需要修正
5. 价格上浮的计算公式：最终单价 = round(标准库单价 × (1 + markup_percent/100), 2)
